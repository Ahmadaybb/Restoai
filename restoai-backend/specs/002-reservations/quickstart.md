# Quickstart — Telegram Table Reservations

**Feature**: `002-reservations` | **Date**: 2026-06-13

This guide explains how to validate the reservation feature end-to-end
locally. It is a *validation* guide — implementation steps live in
`tasks.md`, not here.

## Prerequisites

- Docker stack from `001-takeaway-orders` already running (or rebuild
  with `docker compose up --build`). No new services are added.
- `.env` already configured for `001`; no new required keys.
- The Alembic migration `002_reservations_schema.py` has run (the
  `migrate` service runs `alembic upgrade head` automatically on startup).

## Bring up the stack

```powershell
docker compose up --build
```

The `migrate` service runs `alembic upgrade head` on startup, which
creates the `reservations` table. Check that the table exists:

```powershell
docker compose exec db psql -U restoai restoai -c "\d reservations"
```

`api` is ready when `GET http://localhost:8000/readyz` returns 200.

## US1 — New customer places a reservation (P1 happy path)

1. Open Telegram, start a chat with your bot and type:

   ```
   I'd like to book a table for tomorrow night
   ```

   Expect: the bot detects a reservation intent and begins collecting
   fields. First missing field should be date (or time) — FR-001.

2. Reply with the remaining fields one at a time as the bot asks:
   - Date (e.g., `June 20` or `20/6` — bot reads it back in unambiguous
     format and asks you to confirm before proceeding — FR-009).
   - Time (e.g., `8 PM`).
   - Party size (e.g., `3`).
   - Name (e.g., `Ahmad`).
   - Phone (e.g., `+9613123456`).

3. The bot asks: **Indoor or Outdoor?** Tap **Indoor**. Expect the bot
   to ask: **Smoking or Non-Smoking?** Tap **Non-Smoking** — FR-003,
   FR-004.

4. Expect a confirmation message containing: reference number
   (`RES-XXXXXXX`), date, time, party size, name, phone, and seating
   (indoor, non-smoking) — FR-010, FR-011.

## US2 — Party too large: redirect to call center

1. Start a new chat (or type `/start`). Enter a reservation request and
   provide a party size of **15**.

2. Expect: the bot immediately says it cannot handle groups over 14
   and provides the call center number **1661**. No further fields
   should be asked — FR-007.

## US3 — Terrace size block

1. Start a reservation with party size **6**.
2. When the bot asks seating, choose **Outdoor → Terrace**.
3. Expect: the bot explains the terrace maximum (5 people), offers
   **Outdoor (non-terrace)** or **Indoor** as alternatives — FR-006.
4. Tap **Outdoor (non-terrace)**. Expect a confirmation with
   `seating_preference = outdoor_non_terrace`.

## US4 — Modification (party size change triggers seating re-ask)

1. From a confirmed reservation with terrace seating and party size 3,
   send: `I need to change my reservation, add 3 more people`.
2. The bot finds your reservation by phone number and shows current
   details. Party size is updated to 6.
3. Because 6 > 5 (terrace max), the bot must detect the conflict and
   re-ask seating — FR-015.
4. Tap **Outdoor (non-terrace)**. Expect a re-confirmation with the
   **same reference number** and updated party size and seating — FR-016.

## US5 — Cancellation

1. From a confirmed reservation, send: `Cancel my reservation`.
2. The bot shows the booking details and presents **Yes, cancel** /
   **No, keep it** inline buttons — FR-017, FR-018.
3. Tap **Yes, cancel**. Expect an acknowledgement with the reference
   number — FR-019.

## Multilingual smoke check (Principle III)

- English input → English reservation confirmation.
- Arabic (Lebanese dialect) input (e.g., `بدي احجز طاولة بكرا بالليل
  لـ 4 أشخاص`) → Arabic confirmation message.
- Arabizi input (e.g., `bade ahejez tawle la bukra la 4 ashkhas`) →
  English confirmation message.

Language-routing accuracy on the held-out set must remain ≥ 95% (SC-005,
from `001`; shared classifier).
