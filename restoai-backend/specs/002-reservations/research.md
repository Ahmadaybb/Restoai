# Phase 0 Research: Telegram Table Reservations

**Feature**: `002-reservations` | **Date**: 2026-06-13

This document resolves all technical unknowns and confirms design
decisions before Phase 1 design work begins. Each entry follows the
format: Decision → Rationale → Alternatives Considered.

---

## R1 — ReservationDraft storage: Redis-only vs Postgres snapshot

**Decision**: Redis-only, same as `OrderDraft` (ADR-007 from 001).

**Rationale**: A reservation draft is partial, ephemeral conversation
state. The first durable artifact is the confirmed `Reservation` row in
Postgres. Keeping drafts in Redis eliminates the divergence problem
(Redis and Postgres can't be written in one atomic transaction across
drivers) and keeps the write path simple. The trade-off — losing at most
2 hours of in-flight drafts on a Redis restart — is acceptable because no
money has changed hands and no commitment exists until confirmation.
Conversation turns are still written to Postgres on every turn via the
existing `transcript_repo`, so the audit trail is intact regardless of
draft loss.

**Alternatives considered**: Writing a `reservation_drafts` table to
Postgres in parallel with Redis. Rejected: two-store consistency is more
complex than the problem it solves at this scale.

---

## R2 — Immediate confirmation vs dispatcher review

**Decision**: Immediate confirmation — no dispatcher review step for
reservations.

**Rationale**: A reservation is lower-stakes than an order: no money
changes hands, no POS write path exists, and the restaurant's existing
policy accepts walk-ins regardless. The spec explicitly states "immediate,
no human approval needed." The restaurant reviews their bookings daily and
phones customers for edge cases (operating hours, special events). A
dispatcher gate here would add latency for zero practical benefit given
this policy.

**Alternatives considered**: Queuing reservations in the dispatcher
dashboard (like orders). Rejected by the spec author: explicitly out of
scope.

## R3 — Reference number format

**Decision**: 7-character alphanumeric code, uppercase, prefixed `RES-`
(e.g., `RES-4F2A7B3`). Generated by `secrets.token_hex(4).upper()` (4
random bytes → 8 hex chars, take first 7).

**Rationale**: Short enough to read aloud in a phone follow-up call,
unambiguous (uppercase alphanumeric avoids 0/O and 1/I confusion if the
restaurant system enforces it), and collision probability at expected
booking volumes (low thousands/month) is negligible (1 in ~270 million).

**Alternatives considered**: UUID (too long to quote), sequential integer
(exposes booking volume, requires Postgres sequence coordination).

## R4 — Seating preference as typed enum vs free text

**Decision**: Typed `SeatingPreference` Python enum with four values:
`indoor_smoking`, `indoor_non_smoking`, `outdoor_terrace`,
`outdoor_non_terrace`. Stored as a `varchar(24)` in Postgres.

**Rationale**: The spec has exactly four valid states and the bot drives
the customer through an explicit branching dialog — there is no ambiguity
to resolve at storage time. A typed enum is validatable by Pydantic at
every boundary, and the terrace-block rule (FR-006) can be expressed as a
clean enum comparison rather than string parsing.

**Alternatives considered**: Storing as free-form text with validation in
service logic. Rejected: harder to assert exhaustiveness, easier to
introduce a bug when adding a new seating option in the future.

---

## R5 — Call center number as config vs hardcode

**Decision**: Read from `data/restaurant_info.json`'s
`restaurant.contact.phone` field (value: `"1661"`). Loaded at startup via
the existing zone_repo / menu_repo pattern (load-from-file, LRU cached).

**Rationale**: The number is already present in `restaurant_info.json`
(confirmed by inspection). Centralizing it there means changing the
phone number requires editing one file — not a code change. Hardcoding
violates the spirit of Principle V (config, not code) and would require a
redeploy to update.

**Alternatives considered**: Adding `CALL_CENTER_PHONE` to the `Settings`
pydantic model as an env-var. Rejected: this is restaurant-specific
content, not deployment config. The `restaurant_info.json` is already the
single source of truth for restaurant-side data.

---

## R6 — Modification flow: partial update vs delete-and-recreate

**Decision**: Partial update in place — the existing `Reservation` row is
mutated (date, time, party size, seating) and the reference number is
preserved.

**Rationale**: FR-012 explicitly requires the same reference number to be
used after modifications. Customers quote the reference number when calling
the restaurant for follow-up; changing it mid-modification would break
that. A partial update (SQL UPDATE) is simpler than delete-insert and
keeps the audit trail unambiguous.

**Alternatives considered**: Soft-delete-and-recreate (cancel old, confirm
new). Rejected: produces a new reference number, which the spec disallows.

---

## R7 — Seating re-ask trigger: only party size change vs any modification

**Decision**: Seating is re-asked only when: (a) the customer explicitly
requests a seating change (US3/FR-016b), or (b) the party size change
makes the current seating invalid (terrace with new size > 5). For date
and time changes, seating is preserved without re-asking.

**Rationale**: FR-015 and FR-016b define the two cases precisely. Re-asking
seating for a time-slot change would be annoying UX with no correctness
benefit (seating categories don't change by time).

---

## R8 — Cancellation confirmation gate

**Decision**: Two-step confirmation via an inline `Yes, cancel` / `No,
keep it` button pair (same pattern as order confirm/edit buttons). No
separate "are you sure?" text prompt — the button labels are unambiguous.

**Rationale**: A text prompt ("type YES to confirm") is error-prone for
multilingual customers. Inline buttons have been the proven UX pattern
throughout feature 001. The confirmation dialog shows the full reservation
summary so the customer can verify they're cancelling the right booking.

---

## R9 — Multi-reservation lookup

**Decision**: When a customer has multiple active reservations, the bot
presents a list of reference numbers with summary (date + time) as inline
buttons. The customer taps one; subsequent modification or cancellation
applies to the selected reservation.

**Rationale**: The spec edge case states: "bot lists them all with reference
numbers and asks which one." Inline buttons are the natural implementation.
The expected cardinality is low (most customers have 1 active reservation
at a time) so a simple list is sufficient.

---

## R10 — draft_store reuse vs new reservation_draft_store module

**Decision**: Add a new `app/infra/reservation_draft_store.py` mirroring
`draft_store.py`. Do NOT add reservation keys to the existing
`draft_store.py`.

**Rationale**: The two stores share the same Redis instance and TTL
constant but serve different domain objects. Co-locating them in one file
creates coupling; separate modules keep both files small and independently
reviewable. The key prefix for reservation drafts is `res_draft:{customer_id}`
to avoid collision with `draft:{customer_id}` (order drafts).

## R11 — Reservation date parsing: bot-side vs LLM extraction

**Decision**: Mechanical-tier LLM extraction (same tier as `parse_order`).
The bot sends the customer's raw text to a lightweight `extract_reservation_fields`
tool prompt that returns structured JSON: `{date, time, party_size, name, phone}`.
Informal date formats ("next Friday", "17/6") are returned as ISO-8601
strings after the LLM normalizes them; the bot reads them back for
customer confirmation before proceeding (FR-009).

**Rationale**: Consistent with the two-tier pattern already established in
the codebase. The mechanical tier is cheap and fast for structured
extraction. Using the LLM for normalization avoids shipping a custom date
parser that would need to handle Arabic date names, Lebanese colloquial
time formats ("ba3d bukra"), and mixed-script input.

**Alternatives considered**: Python `dateparser` library. Rejected: adding
a new dependency for a case the existing mechanical LLM tier handles
equally well at zero marginal cost.

## R12 — conversation_service routing: RESERVATION intent

**Decision**: `RESERVATION` is already a member of the `Intent` enum in
`app/domain/language.py` (confirmed by code inspection). The intent
classifier already knows the label. Add a new branch in
`conversation_service.handle_text`:

```python
elif intent_result == Intent.RESERVATION:
    reply_text, buttons = await _handle_reservation_intent(...)
```

**Rationale**: Mirrors the existing `ORDER` / `QUERY` branching pattern.
No changes to the classifier are needed; the label is already trained.

---

## R13 — ReservationDraft field collection: single-shot vs turn-by-turn

**Decision**: Turn-by-turn collection using `chat_state` in Redis (the
existing `draft_store.put_chat_state` / `get_chat_state` infrastructure).
The `chat_state` blob holds a `waiting_for` field (e.g.,
`"reservation_date"`, `"reservation_time"`, `"seating_indoor_preference"`)
that tells the next turn handler which field to collect next.

**Rationale**: The `chat_state` pattern is already implemented in
`draft_store.py`. Using it avoids the complexity of trying to extract all
reservation fields from a single message (customers rarely provide all five
required fields in one go). The bot asks for one missing field at a time,
consistent with the ordering flow.

**Alternatives considered**: Single LLM call to extract all fields at once,
then collect only missing ones. Viable, but the `chat_state` turn-by-turn
approach is already a proven pattern and avoids a more complex state machine.
The `extract_reservation_fields` tool (R11) can still be used on the initial
message to pre-fill any fields the customer provided upfront.
